Source: https://experienceleague.adobe.com/en/docs/authoring-guide/using/home

## &#x200B;1. 메타데이터/주요 사항

### 필수 필드
| 필드 | 요구 사항 | 규칙 |
|---|---|---|
| `title` | 필수 | 최대 60자(영어). 제목 대/소문자 명확성을 위해 필요한 경우가 아니면 제품 이름이 없습니다. 시스템에서 &quot; | ProductName&quot;이 자동으로 표시됩니다. 콜론 또는 대괄호가 포함된 경우 따옴표로 묶습니다. |
| `description` | 필수 | 최대 150~160자. 공백이거나 null일 수 없습니다. 개념은 &quot;다음에 대해 알아보기...&quot;로 시작됩니다. 작업 시작 &quot;방법 알아보기...&quot; 또는 명령형 동사입니다. |

### 선택 사항이지만 자주 사용하는 필드
| 필드 | 유효한 값 | 메모 |
|---|---|---|
| `feature` | 기능 YAML 값과 일치해야 합니다. 제목 대/소문자가 공백(예: &quot;콘텐츠 조각&quot;)과 일치해야 합니다. | 문서당 1-2, 항목에 표시 |
| `feature-set` | 기능 YAML에 대해 검증해야 합니다. | 기능의 상위 응용 프로그램 |
| `role` | 사용자, 개발자, 리더, 관리자, 설계자, 데이터 설계자, 데이터 엔지니어 | 대/소문자를 구분합니다. &quot;Created for&quot;으로 표시됩니다. |
| `level` | 초급, 중급, 고급 | 지정되지 않은 경우 기본값은 초보자 입니다. |
| `solution` | 솔루션 YAML과 일치해야 합니다(예: &quot;Campaign, Campaign Standard v7&quot;). | 여러 값이 허용되며 검색 필터링에 사용됩니다. |
| `topic` | YAML 주제와 일치해야 하며 공백이 있는 제목 대/소문자를 구분해야 합니다. | 제품 간 필터링(예: &quot;통합&quot;) |
| `type` | 설명서, 튜토리얼, 문제 해결 | 저장소 수준, 기본값은 설명서로 설정 |
| `short-description` | 한 문장, 최대 20단어 | 설명이 너무 긴 경우 ExL 랜딩 페이지 |
| `badgePremium` | `label="Premium" type="Positive" url="..."` | 페이지 수준 배지 |
| `mini-toc-levels` | 1-6(기본값 2) | 오른쪽 탐색 제목 표시 깊이 제어 |
| `hidefromtoc` | true/false | 왼쪽 탐색에서 제외하지만 URL에 계속 액세스 |

### 메타데이터 배치
- 문서 수준: YAML 초기 사항으로 Markdown 파일의 상단
- 목차 수준: TOC.md 파일에서
- 저장소 수준: metadata.md 파일

### 주요 서식 규칙
- 쉼표 또는 대괄호가 포함된 값을 따옴표로 묶습니다.
- 쉼표로 구분된 여러 값
- YAML 정의에 대한 대/소문자 구분 일치
- 비어 있거나 비어 있는 태그 값으로 인해 유효성 검사 실패

### 사용되지 않는 필드
seo-title, seo-description, audience, difficulty, uuid (from migration era)

## &#x200B;2. MARKDOWN 구문(ADOBE 기반)

### 텍스트 서식
- 굵게: `**text**`
- 기울임꼴: `*text*`
- 굵은 기울임꼴: `***text***`
- 특수 문자 이스케이프 처리: `# { } [ ] * + - . !` 앞에 백슬래시 `\`을(를) 사용합니다.
- HTML 엔터티: `&lt;` `&gt;` `&amp;` `&mdash;` `&ndash;`

### 제목
- ATX 스타일 사용(`#` 구문 전용, 밑줄/텍스트 스타일 사용 안 함)
- 문서당 H1 1개(일반적으로 메타데이터 `title`과(와) 일치하는 페이지 제목)
- 모든 후속 머리글 H2(`##`) 이하
- 제목 전후에 필요한 빈 라인(첫 번째 제목 제외)
- 최대 69자(영어) / 120자(현지화)
- 최대 ~5단어 기본 설정
- 모든 제목에 대한 문장 대소문자(첫 단어와 고유 명사만 대문자로 표시)
- `title` 메타데이터 필드에 대한 제목 대/소문자만
- 문장 부호 없음(FAQ 머리글에 물음표 사용 가능)
- 사용자 지정 앵커 ID: `## Title {#custom-id}` — 마침표가 아닌 하이픈 사용
- 문서에 중복된 머리글 앵커 ID가 없음
- 예약된 앵커 이름(검색, 결과, 패싯, 페이지 매김, 정렬, 쿼리, 검색 상자, 콘텐츠, 머리글, 바닥글, 기본, 탐색, 사이드바, 페이지, 컨테이너, 래퍼)을 사용하지 마십시오.
- 본문 단락이 하나 이상 있는 모든 제목을 따릅니다(목록/표를 제목 뒤에 직접 배치하지 마십시오).
- 개념 제목: 명사/명사구
- 작업 제목: 명령형 동사(동사 아님 - &quot;세그먼트 만들기&quot; &quot;세그먼트 만들기&quot; 아님)
- 제목 수준 간 병렬 구조

### 목록
- 글머리 기호(순서가 지정되지 않음): `*`, `-` 또는 `+` — 문서 내에서 일관되게 사용
- 주문됨: 모든 항목(자동 번호)에 대해 `1.`
- 목록 앞 및 뒤 빈 줄
- 중첩된 번호 매기기 항목 3개 들여쓰기, 중첩된 글머리 기호 2개 공백
- 글머리 기호 구두점: 끝에 세미콜론/쉼표/연결 생략, 전체 문장에 대해서만 마침표 추가
- 모든 항목이 ≤3단어 또는 UI 레이블/머리글인 경우 마침표를 추가하지 마십시오.

### 링크
- 외부: `[text](https://url.com)`
- 상대(같은 수준): `[text](file.md)`
- 루트 상대: `[text](/help/path/file.md)` — `/`(으)로 시작해야 함
- 딥링크(같은 페이지): `[text](#heading-anchor)`
- 문서 간 딥링크: `[text](file.md#heading-id)`
- 새 탭 강제 적용: `[text](url){target="_blank"}`
- 빈 URL: 클릭 가능하도록 대괄호 `<https://example.com>`에 줄바꿈
- 참조 링크: 절대 링크만 참조 스타일 링크에 대해 작동합니다
- 상대 링크를 통해 여러 TOC 위치에 동일한 파일을 포함하지 않음
- Windows 사용자: 경로에서 백슬래시를 슬래시로 변환

### 이미지
- 기본: `![alt text](image.png "hover tooltip")`
- 크기 조정: `{width="300"}`
- 정렬: `{align="center"}` 또는 `{align="right"}`
- 확대/축소 가능: `{zoomable="yes"}` 또는 `{modal="regular"}`
- 최대 파일 크기: 100MB(5MB 미만 권장)
- 모든 이미지에 대체 텍스트 필요(SEO 및 액세서빌러티)
- 이미지 파일 이름: 하이픈이 있는 소문자
- 지정된 에셋 폴더에 이미지 저장

### 코드 블록
- 인라인: 백틱 `` `code` ``
- 파일 이름, URL, 사용자 정의 용어, 명령
- 펜싱된 코드 블록: 언어 식별자가 있는 트리플 백틱

  ```
  ```javascript

  code here

  ```

  ```

  
  
- 옵션: `{line-numbers="true"}`, `{start-line="7"}`, `{highlight="11-13, 16"}`
- 펜싱된 코드 블록에 항상 언어 식별자 포함

### 표
- 구분 기호 행에는 열당 하이픈이 3개 이상 필요합니다. `|---|---|---|`
- 정렬: 왼쪽 `|---|`, 가운데 `|:---:|`, 오른쪽 `|---:|`
- 리터럴 파이프 이스케이프 처리: `\|` 또는 `&vert;` 사용
- HTML 테이블 허용, `<table style="table-layout:auto">` 또는 `table-layout:fixed` 사용
- HTML 테이블 정렬: `<td align="left|center|right">`
- HTML 테이블 내에서 Markdown 구문을 사용하지 마십시오(예: HTML 테이블에서 `[!NOTE]`이(가) 작동하지 않음).
- 테두리 제거: `<tr style="border: 0;">`
- Markdown 표 레이아웃 옵션: 빈 줄이 있는 표 뒤에 `{style="table-layout:auto"}`을(를) 추가합니다.
- 가로 스크롤 막대 가시성 문제로 인해 매우 넓은/높은 테이블을 사용하지 마십시오.

## &#x200B;3. 특수 ADOBE 구문 확장

### 설명선 / 경고

```
>[!NOTE]
>
>Text here.

>[!TIP]
>
>Text here.

>[!IMPORTANT]
>
>Text here.

>[!WARNING]
>
>Text here.

>[!CAUTION]
>
>Text here.

>[!ADMIN]
>[!AVAILABILITY]
>[!PREREQUISITES]
>[!INFO]
>[!ERROR]
>[!SUCCESS]
```

- 중요: `>`에서 `[ !` 사이의 공백 없음 — `> [!NOTE]`이(가) 아닌 `>[!NOTE]` 사용
- `>[!NOTE]`과(와) 본문 텍스트 줄 사이에 빈 줄 추가

### 탭

```
>[!BEGINTABS]

>[!TAB iOS]
Content here

>[!TAB Android]
Content here

>[!ENDTABS]
```

### 접을 수 있는 섹션(아코디언)

```
+++Click to expand
Content inside
+++
```

참고: 중첩된 축소 가능 섹션은 지원되지 않습니다.

### 음영 상자

```
>[!BEGINSHADEBOX "Optional Title"]
Content here
>[!ENDSHADEBOX]
```

### 비디오

```
>[!VIDEO](https://video.tv.adobe.com/v/ID/?quality=12&learn=on)
```

성적 증명서용 `{transcript=true}`을(를) 추가합니다.

### 다음과 같음

```
>[!MORELIKETHIS]
>* [Article 1](url)
>* [Article 2](url)
```

### 상황별 도움말

```
>[!CONTEXTUALHELP]
>id="..."
>title="..."
>abstract="..."
>additional-url="..."
```

### 배지(인라인)

```
[!BADGE Label]{type=Informative url="https://example.com" tooltip="text"}
```

형식: `Informative`(파란색), `Positive`(녹색), `Negative`(빨간색), `Neutral`(회색), `Caution`(노란색)

### 텍스트 강조 표시(미리 보기)

```html
<span class="preview">highlighted text</span>
```

### 및 코드 조각 포함
- 전체 파일 포함: `{{$include /help/_includes/legal-blurb.md}}`
- 코드 조각(머리글 없음): `{{snippet-id}}`
- `help > _includes/` 폴더에 재사용 가능한 파일 저장
- `help > _includes/snippets.md`에 저장된 코드 조각
- 포함 은 제목을 가질 수 있고 스니펫은 가질 수 없습니다.
- 동일한 저장소 내에서만 작동합니다(교차 저장소 아님).
- 참조되지 않는 파일에서 `{{` 또는 `}}`자 이스케이프 처리

### DNL(현지화하지 않음) 태그
- `[!DNL ProductName]` — 번역할 수 없는 제품/브랜드 이름을 래핑합니다.
- 페이지 내 첫 번째 또는 주요 언급 사용

### UICONTROL 태그
- `[!UICONTROL Button Label]` — UI 요소 텍스트(단추, 메뉴, 필드 이름)를 래핑합니다.
- 기계 번역이 아닌 제품 데이터베이스에서 UI 문자열을 가져옵니다.
- UICONTROL 태그 내에 아포스트로피, 따옴표 또는 구두점 없음
- 단계에서 참조된 UI 요소에 굵은 서식 사용

### 지원되지 않는 기능
- 작업 목록(확인란)
- 이모지
- 가로 규칙
- 중첩된 축소 가능 섹션

## &#x200B;4. 파일 이름 지정 및 폴더 구조

### 파일 이름 지정 규칙
- 하이픈만 사용하는 소문자 파일 이름
- 대문자, 밑줄, 마침표 또는 공백 없음
- 설명적이고 SEO에 친숙한 이름(예: `cmo.md`이(가) 아닌 `calculated-metric-overview.md`)
- 파일 이름에서 숫자를 사용하지 마십시오. 수사적 단어를 사용하십시오.
- 예약된/충돌하는 이름 방지: `metadata.md`, `search.md`
- 라이브 파일의 이름을 변경하려면 리디렉션 관리(URL 변경)가 필요합니다.

### 폴더 구조
- 디렉터리/폴더 이름은 URL에 영향을 주지 않습니다. 저장소 이름과 파일 이름만 URL에 영향을 줍니다.
- TOC.md는 Git 폴더 위치가 아니라 탐색 계층 구조를 제어합니다
- 지정된 에셋 폴더에 이미지/에셋 저장
- `help/_includes/`에 저장된 포함 내용 재사용 가능
- `help/_includes/snippets.md`에 저장된 코드 조각

### TOC.md 규칙
- 모든 문서는 Experience League에서 렌더링하려면 TOC.md에 나열되어야 합니다
- H1 형식: `# Guide Title {#anchor-id}` — 앵커에서 기본 URL 경로를 생성합니다.
- 섹션 머리글에는 앵커 ID가 있어야 합니다. `+ Section Name {#section-id}`
- 섹션 머리글(상위 항목)은 링크 자체일 수 없습니다.
- 문서 링크: `+ [Title](path/file.md)`
- 섹션 앵커 ID를 변경하면 모든 중첩된 문서 URL이 변경됩니다(리디렉션 필요)
- 전체(`+`, `*` 또는 `-`)에서 일관된 글머리 기호 스타일 사용
- 목차 메타데이터: `user-guide-description`, 선택적 `breadcrumb-title`
- `mini-toc-levels`: 오른쪽 탐색 제목 표시를 제어합니다(1-6, 기본값 2).

## &#x200B;5. 콘텐츠 품질 및 편집 표준

### 음성 및 톤
- 수동보다 능동적인 음성 선호
- 현재 시제 후 미래 시제
- 지침 콘텐츠에 대한 두 번째 사람(&quot;귀하&quot;)
- 35단어 미만의 문장
- 강하고 정확한 동사 — 약한 부사(&quot;매우&quot;, &quot;극히&quot;)를 피하십시오.

### 단계 / 절차
- 각 단계는 하나의 간결한 명령(한 문장)입니다
- 추가 정보는 다음 줄에 표시됩니다(단계 아래에 들여쓰기)
- 명령형 동사로 각 단계를 시작하십시오.
- 절차를 약 7단계로 제한(하위 작업으로 나누기 전 최대 10단계)
- 단계 앞에 개념 정보 배치
- 관련 단계 후에 스크린샷 배치
- 판독기 방향에 대해 페이지/탭 이름 반복
- 단계의 UI 요소에 대한 굵게 UICONTROL 서식

### 인터페이스 용어
- **열기/닫기**: 응용 프로그램 및 프로그램
- **이동**: 메뉴, 탭 또는 UI 위치
- **목록에서 텍스트, 셀 또는 옵션 선택**
- **Click**: 마우스 동작(필수 항목이 아닌 경우 컨트롤 형식을 지정하지 않음)
- **드롭다운 메뉴**(&quot;드롭다운 목록&quot; 아님)

### 제품 이름
- 명확성이 요구되지 않는 한 제목/제목에 제품 이름을 추가하지 마십시오.
- 제품 이름 앞에 &quot;the&quot; 생략
- 안내서 수준에서 첫 번째 언급에 &quot;Adobe&quot; 포함(상표 요구 사항)
- 법적으로 필요하지 않은 경우 보조 언급에서 &quot;Adobe&quot;를 삭제하십시오
- 오역을 방지하기 위해 제품 이름에 DNL 태그 사용

### 강조 및 서식
- 굵게: UI 요소 및 키보드 작업
- 기울임체: 오류 메시지, 외국어, 강조
- 코드(백틱): 파일 이름, URL, 사용자 정의 용어
- UI 요소 주위에 따옴표 없음(대신 UICONTROL 태그 사용)

### 대문자 사용
- 제목, 탐색, 소호에 대한 문장 케이스
- `title` 메타데이터 필드에 대한 제목 대/소문자만
- 고유 명사는 항상 대문자로 표기됩니다.

## &#x200B;6. SEO 우수 사례

- 페이지당 하나의 H1 — 간결하고 설명적이며 페이지에 대한 내용을 사용자에게 알려주어야 합니다.
- 순차적 머리글 계층(h1 → h2 → h3, 건너뛰지 않음 수준)
- H1, 본문 및 메타데이터에 키워드 포함(`keywords` 메타 태그가 아님 - Google에서 무시)
- 키워드 스터핑 방지(의도 > 빈도)
- 제목: 최대 50-60자, 시스템 추가 &quot;| Adobe ProductName&quot; 자동
- 설명: 150-160자, call to action이어야 하며 콘텐츠를 반복해서는 안 됩니다
- 모든 이미지에 설명 대체 텍스트 추가
- 비디오 트랜스크립트 포함(검색 엔진은 텍스트만 읽을 수 있음)
- 관련 페이지에 전략적으로 연결(고립된 페이지 방지)
- 구조화된 요소 포함: 번호 매기기 목록, 하위 머리글, 코드 블록
- AnswerThePublic, Google Trends와 같은 도구를 사용하여 키워드 검색
- 콘텐츠는 E-E-A-T(경험, 전문 지식, 권한, 신뢰성)를 보여 주어야 합니다.

## &#x200B;7. 현지화

### 기계 번역 먼저
- 영어 소스 컨텐츠는 현지화된 버전을 자동으로 생성합니다.
- 지원되는 언어: 독일어, 프랑스어, 일본어, 이탈리아어, 스페인어, 포르투갈어(브라질), 중국어 간체, 중국어 번체, 한국어, 네덜란드어, 스웨덴어

### 로컬라이제이션을 위한 작성
- 전반에서 일관된 용어
- 짧은 문장 및 능동적인 음성
- 표준 영어 단어 순서(제목 → 동사 → 개체)
- 상대 대명사(&quot;that&quot;, &quot;which&quot;) 사용
- 피하십시오: 유머, 숙어, 전문 용어, 지역 관용구, 은유, 불필요한 단어

### 로컬라이제이션 태그
- `[!UICONTROL Label]` — UI 문자열이 기계 번역이 아닌 제품 DB에서 가져오도록 합니다.
- `[!DNL ProductName]` — 제품/브랜드 이름이 번역되지 않도록 합니다.
- 현지화하지 않음 폴더의 이미지는 현지화에서 제외됩니다

## &#x200B;8. 콘텐츠 유형

- **안내서**: TOC에서 참조된 페이지의 온라인 컬렉션입니다. 공식 모범 사례를 다룹니다.
- **자습서**: 특정 사용 사례에 대한 지침 콘텐츠(비디오 또는 텍스트), 학습 저장소에 게시됨
- **참조 안내서**: 개발자 API/SDK; 일반적으로 테이블 형식; developer.adobe.com에 게시됨
- **과정**: 특정 사용자 역할을 대상으로 전문가가 선별한 학습 모음입니다.
- **기술 자료 문서**: 간단하고 일시적으로 관련된 문제 해결 콘텐츠
- **랜딩 페이지/홈 페이지**: 별도 관리(SCCM)

## &#x200B;9. 피해야 할 일반적인 유효성 검사 오류

- 누락되었거나 빈 `title` 또는 `description` 메타데이터
- `description`이(가) &quot;자세히 알아보기...&quot;로 시작하지 않음 또는 &quot;방법 알아보기...&quot;
- 호출 구문(`>[!NOTE]` 대신 `> [!NOTE]`)의 `>`과(와) `[!` 사이의 공간
- 굵은 표식의 공백: `**text **`(후행 공백은 굵게 구분됨)
- HTML 표 내의 Markdown 구문(예: 설명선이 여기에서 작동하지 않음)
- 문서에 머리글 앵커 ID 복제
- 마침표를 포함하는 앵커 ID(대신 하이픈 사용)
- 예약된 앵커 이름(검색, 결과, 머리글, 바닥글 등) 사용
- YAML 정의와 일치하지 않는 `role`, `level`, `feature`, `topic` 값
- 본문 텍스트가 없는 스택 머리글
- H1이 문서당 두 번 이상 사용됨
- 건너뛴 제목 수준(예: H1 → H3)
- 대문자, 밑줄 또는 공백으로 파일 이름 지정
- 이미지에 대체 텍스트 누락
- 동명사 작업 제목(&quot;생성 중...&quot;) 대신 &quot;만들기...&quot;)
